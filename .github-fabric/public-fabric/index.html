About

What Is GitHub Fabric?

GitHub Fabric starts with a stubbornly simple premise: GitHub isn't only a place to store code — it can be the place code becomes a system. Not "CI for a repo," but repos as first-class, runnable infrastructure units.

In a Fabric you don't just reference upstream projects. You ingest them, splice in curated changes, and run them — reliably, repeatedly, and by name — on top of GitHub Actions.

Architecture — The Three Planes

1 · Source Plane

Faithful capture of upstream repos. Track origin + commit SHA; keep raw imports segregated from runtime material. The source plane is for truth, not convenience.

2 · Transformation Plane

Cull bulk, patch configs, normalize entry-points, and wrap modules with an adapter contract — turning heterogeneous repos into consistent, driveable units.

3 · Execution Plane

GitHub Actions is the runtime substrate: scheduled syncs, workflow-dispatch runs by name, contract tests, artifact provenance, caching, and isolation.

Upstream DNA

Repos as Genetic Material

Most platforms treat upstream code like a dependency. GitHub Fabric treats it like genetic material — something you import, normalize, and express under controlled conditions.

  • Taking in source from one or many upstream repos
  • Splicing targeted edits — policy, structure, configs, adapters
  • Expressing a runnable form via a standardized entry-point
  • Preserving provenance — exact upstream commit, applied patches, and outputs
Workflow Lifecycle

Resolve → Materialize → Validate → Execute → Emit

Resolve Materialize Validate Execute Emit

Every run converts a module name into a spec, fetches upstream, applies cull + patch rules, stamps provenance, runs contract tests, executes with constrained permissions, and publishes artifacts.

Name Addressability

Modules

Module names map to runnable capability. Trigger run <name> and the Fabric resolves code, rules, environment, and workflow — then executes.

Templates

Template names map to scaffolded new repos. The Fabric becomes a production line: standard workflows, sane defaults, consistent structure.

Why It Matters

Curated Distribution

Scattered capability becomes a shelf of approved, runnable modules.

Reviewable Upgrades

Upstream changes arrive as PRs with diffs, test results, and pinned SHAs.

Reproducibility by Design

Runs are tied to exact inputs and versions. "It broke" is always traceable.

Unified Governance

Approvals, policies, audit trails, and secrets management live where the work executes.
