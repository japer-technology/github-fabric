# Fabric Mimic

> An analysis of .github-mimic—what it means when a repository compiles its own command system, and how repo-local intelligence that infers intent and produces versioned operational surfaces relates to the Fabric's architecture.

---

## 1. What .github-mimic Actually Proposes

The `.github-mimic` concept document describes a pattern that sounds, at first reading, like another AI-assisted automation framework. Drop a folder into a repository. An LLM reads the repository. It produces helpful commands. Users invoke those commands through issues.

Read more carefully, the proposal is structurally different from conventional AI tooling in three specific ways:

**First**, the intelligence is repo-local. The `.github-mimic` folder is not a SaaS integration, not an external service, not a marketplace action that calls home. It is a directory committed to the repository, versioned with the repository, governed by the same branch protections and review processes as every other file. The intelligence *is* the repository, not something attached to it.

**Second**, the LLM's role is compiler, not assistant. The system does not respond conversationally to ad hoc questions. It observes the repository's actual structure—CI workflows, build scripts, dependency manifests, PR patterns, contribution conventions—and *compiles* that understanding into an explicit command surface: versioned YAML specs that describe what operations exist, what inputs they accept, what they are allowed to modify, and what evidence they must produce. The LLM is a compiler whose source language is "repository reality" and whose target language is "executable command specifications."

**Third**, the system is multiplexed through lenses. A lens is not a persona or a prompt template. It is a policy-plus-objective bundle that changes which signals are prioritized, what outputs are produced, what safety gates apply, and what "success" means. The same repository, observed through the `security` lens, produces different commands, different risk assessments, and different governance requirements than the same repository observed through the `docs` lens. The repository does not have one command surface—it has as many as it has lenses, each governed independently.

These three properties—repo-locality, command compilation, and multiplexed lenses—distinguish `.github-mimic` from both conventional CI automation and conversational AI assistants. The question is what they mean for the Fabric.

## 2. The Fabric Connection: Observation as a Missing Layer

### 2.1 What the Fabric Already Does

The Fabric's architecture is organized into three planes:

- **Source Plane**: Ingest upstream repositories as raw material.
- **Transformation Plane**: Splice, cull, and normalize the ingested code into governed modules.
- **Execution Plane**: Run the modules through GitHub Actions with full provenance and auditability.

This pipeline converts external code into governed, reproducible, name-addressable capability. It answers the question: *given an upstream project, how do we make it runnable, auditable, and composable inside our repository?*

### 2.2 What the Fabric Does Not Yet Do

The Fabric's planes are mechanistic. They describe *how* to process code, but they do not infer *what the code is trying to do*. The Source Plane clones a repository; it does not understand what the repository's build system expects. The Transformation Plane splices configuration; it does not infer the repository's release conventions or test philosophy.

This is where `.github-mimic` occupies a gap. Its Observation Layer—the mechanism that reads build scripts, CI workflows, dependency manifests, release tags, contribution templates, and PR history—is precisely the inference step that the Fabric's planes skip. The Fabric assumes that a human has already understood the upstream and encoded that understanding in splice rules. `.github-mimic` proposes that the understanding itself can be automated.

### 2.3 Observation as a Fourth Plane

If we map `.github-mimic`'s architecture onto the Fabric's planes, the relationship becomes clear:

| .github-mimic Layer | Fabric Analog | Relationship |
|---|---|---|
| **Observation Layer** | *(No direct analog)* | New: infers repository mechanics |
| **Model Layer** | Source Plane | Parallel: both maintain representations of repository state |
| **Compilation Layer** | Transformation Plane | Parallel: both convert representations into executable form |
| **Execution (via Issues/Actions)** | Execution Plane | Identical substrate |

The Observation Layer is the new element. It fills the gap between "we have the code" and "we understand what the code does." In Fabric terms, it is a fourth plane—an **Inference Plane**—that sits before Source and informs how Transformation should proceed.

This is not a theoretical distinction. When the Fabric ingests a new upstream project, the most labor-intensive step is understanding the project's build system, test conventions, release process, and operational invariants. If `.github-mimic`'s observation mechanism can automate even a fraction of that understanding, it reduces the human effort required to bring a new upstream into the Fabric.

## 3. Command Compilation: Why Compiling Is Different From Executing

### 3.1 The Conventional Model

Most AI-assisted development tools follow a request-response pattern: the user asks a question, the LLM generates an answer, the user evaluates it. The LLM is a reactor—it responds to stimuli.

This model works for ad hoc tasks but fails to produce durable operational surfaces. The LLM's output is a transient response, not a versioned artifact. The knowledge it encodes evaporates after the conversation ends. The next user who asks a similar question starts from scratch.

### 3.2 The Compilation Model

`.github-mimic` inverts this by treating the LLM as a compiler:

```
Source Code:      Repository artifacts (CI, scripts, configs, conventions)
       │
   Compiler:      LLM (inference + structured output)
       │
Target Code:      Command specifications (YAML, versioned, reviewable)
```

The output is not a chat message. It is a set of durable artifacts:

- **repo-profile.json**: The inferred purpose, technology stack, constraints, and invariants of the repository.
- **commands/*.yaml**: Explicit command specifications with inputs, constraints, verification steps, and output types.
- **lenses/*.yaml**: Lens policies that define how each operational perspective interprets the repository.
- **memory/*.md**: Accumulated decisions, known pitfalls, and do/don't lists.

These artifacts are committed to git. They are reviewable. They are diffable. They evolve with the repository. And critically, they can be *edited by humans*—a developer who disagrees with an inferred convention can simply edit the spec file and commit the correction.

### 3.3 Why This Matters for the Fabric

The Fabric already produces durable artifacts from upstream code: splice rules, lifecycle scripts, agent personas, state conventions. `.github-mimic`'s command compilation is the same pattern applied to a different input domain. Instead of compiling upstream code into Fabric modules, it compiles repository *behavior* into executable command specifications.

The implication is that `.github-mimic` and the Fabric could share a compilation architecture. The Fabric's Transformation Plane already has the machinery for converting structured input into governed output. `.github-mimic`'s Compilation Layer is a specialization of that same machinery, applied to inferred repository semantics rather than ingested source code.

## 4. Lenses: Multiplexing Repository Understanding

### 4.1 What a Lens Is

The concept document defines a lens as an operating mode that changes:

- Which signals are prioritized
- What outputs are produced
- What safety gates apply
- What "success" means

The proposed lenses—developer, maintainer, security, operations, documentation, product—are not personas in the chatbot sense. They are *policy bundles* that constrain and direct the system's behavior.

### 4.2 Lenses as Governance Instruments

Consider two commands issued against the same repository:

```
/mimic lens docs: sync and update reference docs for recent changes
/mimic lens security: review workflow token permissions and propose least-privilege PR
```

The first command operates under the `docs` lens, which has:
- Default risk: **low**
- Allowed modifications: documentation files
- Safety gate: doc build/test if available
- Success: docs accurately reflect current code

The second command operates under the `security` lens, which has:
- Default risk: **high**
- Allowed modifications: workflow files, permission configurations
- Safety gate: mandatory human approval from CODEOWNERS and SECURITY_TEAM
- Success: permissions reduced without breaking CI

The same underlying intelligence—the LLM's understanding of the repository—is filtered through different governance constraints. The lens is not changing what the system *knows*. It is changing what the system is *allowed to do* and what *evidence it must produce*.

This maps directly to the Fabric's governance model. The Fabric already enforces different constraints on different operations: scoped permissions, sentinel guards, fail-closed behaviors. Lenses formalize this into a named, configurable policy layer. A lens is, in Fabric terms, a **governance profile** that can be selected per invocation.

### 4.3 The Combinatorial Surface

Six lenses applied to a repository with, say, twenty inferred commands produce 120 potential lens-command combinations. Not all are meaningful—a `product` lens applied to a `run-linter` command may add no value. But the combinatorial surface is real, and it means that the command system grows faster than the number of individual commands.

This creates both opportunity and risk. The opportunity is comprehensive coverage: every operational concern (security, documentation, operations) has a systematic entry point for every capability. The risk is sprawl: an explosion of combinations that are difficult to test, validate, and maintain.

The Fabric's response to this should be the same principle it applies elsewhere: governance through versioned artifacts. If every lens-command combination produces a reviewable spec, the combinatorial surface is manageable because it is auditable.

## 5. Process Portability: The Most Ambitious Claim

### 5.1 The Claim

Section 4 of the concept document makes the most ambitious claim in the proposal:

> Many coordination and orchestration approaches can be imported as textual process descriptions (rules, constraints, control loops), then compiled into repo-native commands, rather than embedding heavyweight orchestration libraries.

Read plainly, this says: you do not need to install a framework to get its behavior. You can describe the framework's behavior in text, feed that description to the LLM compiler, and produce repo-native commands that implement the same behavior without the framework's runtime.

### 5.2 The Mechanism

The proposed mechanism is straightforward:

1. Capture the process description (an algorithm, a set of policies, role definitions, a control loop narrative).
2. Feed it to the LLM as part of the compilation context.
3. Produce a command spec with explicit inputs, checks, and outputs that implements the described process.
4. Version the process description alongside the command spec so the provenance is clear.

### 5.3 Connection to the Fabric's DNA Metaphor

The Fabric already practices a version of process portability. When it ingests an upstream project like OpenClaw, it does not adopt OpenClaw's orchestration model wholesale. It extracts the capabilities, normalizes them behind the Fabric's invocation surface, and discards the runtime model. OpenClaw's persistent WebSocket connections become issue-triggered GitHub Actions jobs. OpenClaw's multi-channel integration narrows to the Issues channel. The *behavior* is preserved; the *infrastructure* is replaced.

`.github-mimic` generalizes this pattern. Instead of importing code and replacing infrastructure, it imports *process descriptions* and compiles them into commands. The process description is the "genetic material"—the textual DNA that encodes behavior. The compilation step is the "expression"—the Fabric's Transformation Plane, now applied to methodology rather than source code.

### 5.4 Honest Assessment

This claim deserves scrutiny. Not all processes can be reduced to text and compiled into commands without loss. Processes that depend on:

- Real-time human judgment at unpredictable decision points
- External state that cannot be observed from the repository
- Timing-dependent interactions between multiple actors
- Tacit knowledge that has never been articulated

...will resist compilation. The concept document's honesty about this limitation—"ambiguous repo intent may require early profiling runs and human correction"—is appropriate. Process portability is not universal. It is a useful pattern for well-documented, rule-based coordination processes. For messy, judgment-laden processes, it is a starting point that requires human refinement.

## 6. Reliability: Multiplexed Decision Trees

### 6.1 The Problem

LLMs are probabilistic. They produce plausible outputs that are usually correct but sometimes wrong. For low-stakes operations—generating documentation, summarizing an issue—occasional errors are correctable and tolerable. For high-stakes operations—modifying CI workflows, changing security configurations, proposing dependency updates—a single error can propagate through the repository's automation and cause real damage.

### 6.2 The Solution: Redundancy as a Substitute for Certainty

`.github-mimic` proposes multiplexed decision trees: for risky actions, re-derive key facts through multiple independent reasoning paths.

- **Restatement**: Have the system restate the request in its own words and extract constraints.
- **Invariant checking**: Explicitly ask "what must not change?" and verify that the proposed action preserves those invariants.
- **Adversarial failure search**: Ask "how could this break?" and search for failure modes.
- **Cross-sample agreement**: Optionally, run the same reasoning multiple times (or with multiple models) and compare outputs.

This is not a new idea in reliability engineering. It is the same principle behind redundant flight computers, N-version programming, and Byzantine fault tolerance: when you cannot guarantee that any single computation is correct, require agreement between independent computations.

### 6.3 Two-Phase Commit Behavior

The reliability model extends to execution through a two-phase commit:

| Phase | Actions | Gate |
|---|---|---|
| **Phase 1: Propose** | Generate plan, create PR, attach evidence | Automated checks pass |
| **Phase 2: Verify** | Run CI, request human approvals, check policies | All gates pass |
| **Apply** | Merge PR | Only after Phase 2 succeeds |

This is the Fabric's existing PR-based governance model, formalized. The Fabric already requires that agent changes flow through PRs with CI validation. `.github-mimic` makes this two-phase structure explicit and attaches evidence requirements to each phase.

### 6.4 The Evidence-First Standard

The concept document's strongest governance principle is also its simplest: "No 'trust me' changes: every claim is paired with logs, diffs, or checks."

This principle aligns perfectly with the Fabric's provenance model. The Fabric commits every action—every reply, every file change, every memory update—with a hash, an author, and a diff. `.github-mimic` extends this from actions to *claims*: if the system asserts that a change is safe, it must show the evidence. If it asserts that a process was followed, it must produce the logs.

In the Fabric's architecture, this is the evidence-first standard operationalized as a pipeline requirement: no PR produced by `.github-mimic` can be opened without an attached evidence bundle.

## 7. The Issue-Driven UX: High-Latency, High-Value

### 7.1 Why Issues, Not Chat

`.github-mimic` is explicitly designed for invocation through GitHub Issues, not through a conversational chat interface. This is a deliberate design choice that the concept document justifies through the constraint environment:

- GitHub Actions has non-trivial latency (30 seconds to minutes per invocation).
- Actions minutes have real cost.
- Every invocation should produce durable, auditable artifacts.

These constraints push toward fewer, more deliberate invocations—the opposite of a rapid back-and-forth chat. An issue is a durable artifact with its own lifecycle: it can be labeled, assigned, referenced, linked to PRs, and searched. A chat message is ephemeral.

### 7.2 The Invocation Surface

The proposed command patterns reveal the design intent:

```
/mimic status                    → report on system state
/mimic profile                   → infer and display repo understanding
/mimic lens docs: <instruction>  → execute under a specific lens
/mimic command list              → enumerate available commands
/mimic run <command> with <inputs> → execute a compiled command
```

This is a CLI embedded in an issue tracker. Each command is a discrete, auditable invocation. Each response is a permanent comment on a permanent issue. The conversation history is the audit trail.

### 7.3 Relationship to the Fabric's Issue-Driven Model

The Fabric already uses issues as its primary invocation surface. The GMI system—the Fabric's minimal viable agent—is triggered by issue events and responds through issue comments. `.github-mimic` extends this pattern with a structured command syntax that is more explicit than free-form issue text.

The distinction is between *natural language invocation* (write whatever you want in an issue) and *structured invocation* (use a command prefix with a defined syntax). The Fabric's current model is closer to natural language; `.github-mimic` proposes a command surface. Both are valid. The command surface has the advantage of being machine-parsable and less ambiguous; the natural language surface has the advantage of lower barrier to entry.

A mature system likely supports both: the command surface for power users and automation, and a natural language fallback for ad hoc requests.

## 8. Security and Governance: The Threat Model

### 8.1 Attack Surfaces

The concept document identifies the key attack surfaces with clarity:

- **Prompt injection via issues**: A malicious user crafts an issue that manipulates the LLM into executing unintended actions.
- **Malicious PR content**: Code submitted through PRs that exploits the system's automated processing.
- **Dependency scripts**: Compromised dependencies that execute during build or install steps.
- **Workflow token overreach**: Actions workflows with overly broad permissions.
- **Exfiltration via logs/artifacts**: Sensitive data leaked through workflow outputs.

### 8.2 Mitigations

The proposed mitigations are sound and align with the Fabric's existing security model:

| Mitigation | Fabric Analog |
|---|---|
| Least-privilege `GITHUB_TOKEN` | Fabric's scoped permissions model |
| Explicit allowlists of modifiable paths | Fabric's splice rules limiting agent scope |
| Mandatory PR-based changes | Fabric's PR-only governance |
| No secrets in model prompts | Fabric's separation of reasoning from credentials |
| Sandboxing for untrusted code | Fabric's job isolation and sentinel guards |
| Risk-tiered approvals | Fabric's collaborator-gated invocation |

The alignment is not coincidental. `.github-mimic` and the Fabric share the same execution substrate (GitHub Actions) and the same governance substrate (GitHub's permission model). The security models converge because the constraints are the same.

### 8.3 The Unique Risk: Command Surface Integrity

`.github-mimic` introduces a risk that the Fabric's current architecture does not face: **command surface poisoning**. If the compiled command specifications are versioned in the repository, and the LLM can update them, then a compromised LLM inference could alter command specs to weaken safety gates, expand allowed paths, or reduce approval requirements.

The mitigation is the same two-phase commit: command spec changes flow through PRs, require review, and are subject to CI validation. But the threat is worth naming explicitly because command specs *define* the system's behavior—a change to a command spec is not an ordinary code change. It is a change to the system's operational rules.

The Fabric's response should be to treat the `commands/` and `lenses/` directories as high-sensitivity paths, requiring CODEOWNERS approval for any modification, regardless of the risk tier of the command that produced them.

## 9. What .github-mimic Means for the Fabric

### 9.1 Extension, Not Replacement

`.github-mimic` does not replace the Fabric. It extends it. The Fabric provides the infrastructure: workflows, governance, provenance, execution. `.github-mimic` provides the observation and compilation: understanding what the repository does and producing commands that mirror its capabilities.

In architectural terms:

```
┌─────────────────────────────────────────────────────────┐
│                     .github-mimic                        │
│                                                         │
│  Observation Layer → Model Layer → Compilation Layer     │
│  (infer reality)    (represent)   (produce commands)     │
└─────────────────────────┬───────────────────────────────┘
                          │
                    Command Specs
                    (versioned YAML)
                          │
┌─────────────────────────┴───────────────────────────────┐
│                    GitHub Fabric                          │
│                                                         │
│  Source Plane → Transformation Plane → Execution Plane    │
│  (ingest)      (splice/normalize)     (run via Actions)  │
└─────────────────────────────────────────────────────────┘
```

`.github-mimic` sits above the Fabric and produces the artifacts that the Fabric's Execution Plane consumes. The command specs are the interface between them.

### 9.2 The Packaging Question

Should `.github-mimic` be a standalone package or part of the Fabric?

Arguments for standalone:

- It can be useful in repositories that do not use the Fabric at all. Any GitHub repository with Actions can benefit from an inferred command surface.
- It has a simpler installation story: drop one folder, commit, done.
- It maintains focus—repo observation and command compilation—without coupling to the Fabric's module ingestion machinery.

Arguments for integration:

- The governance models are identical. Maintaining two parallel governance implementations is wasteful.
- The Fabric's existing lifecycle (resolve → materialize → validate → run → emit) can accommodate `.github-mimic`'s compilation step as a new phase.
- The Fabric's module registry could expose `.github-mimic`-compiled commands alongside Fabric modules, creating a unified invocation surface.

The pragmatic answer is: standalone first, integration later. `.github-mimic` as a drop-in folder that works without the Fabric validates the concept with the lowest barrier to entry. Integration into the Fabric follows when the patterns stabilize and the interface between observation and execution is well-understood.

### 9.3 What the Fabric Gains

If `.github-mimic` succeeds, the Fabric gains several properties it does not currently possess:

**Automated repo profiling.** When the Fabric ingests a new upstream project, `.github-mimic`'s Observation Layer can generate the initial repo profile—technology stack, build system, test conventions, release patterns—reducing the manual effort required to write splice rules.

**Self-updating command surfaces.** As the repository evolves, `.github-mimic` can re-observe and re-compile, updating command specs to reflect new capabilities, changed conventions, or deprecated patterns. The command surface co-evolves with the code, rather than requiring manual maintenance.

**Multi-lens governance.** The Fabric currently applies governance uniformly. `.github-mimic`'s lens model allows different governance profiles for different operational concerns, enabling fine-grained risk management that the Fabric's current binary (allow/deny) model does not support.

**Evidence-based trust.** The evidence-first standard gives the Fabric a mechanism for graduated trust: a command that consistently produces valid evidence and passes gates can be promoted to lower review requirements, while a command with a history of failures requires more scrutiny. Trust is earned, not configured.

## 10. Relationship to Previous Analyses

### 10.1 Implications (.ANALYSIS-Implications.md)

The Implications analysis argues that repositories are operating systems for AI agents. `.github-mimic` extends this metaphor: if the repository is the operating system, then the compiled command surface is the **shell**—the user-facing interface through which the operating system's capabilities are invoked.

The Implications document also describes the Fabric's governance dividend: agent behavior constrained by the same mechanisms that govern human contributions. `.github-mimic` deepens this dividend by making the constraints themselves versioned and reviewable. Governance is no longer just a property of the repository's GitHub settings—it is encoded in command specs and lens policies that can be diffed, reviewed, and improved.

### 10.2 Persistent Server Intelligence (.ANALYSIS-Persistent-Server-Intelligence.md)

The Persistent Server analysis examines what happens when the agent provisions its own infrastructure. `.github-mimic`'s command compilation is the cognitive analog: instead of provisioning a server to extend its *execution* capability, the agent compiles a command surface to extend its *operational* capability.

The self-optimization loop described in the Persistent Server analysis—observe, design, provision, consume, refine—maps directly to `.github-mimic`'s compilation cycle: observe the repository, compile command specs, execute commands, observe results, refine specs. The pattern is the same; the domain shifts from infrastructure to operations.

### 10.3 Persistent Local Server Intelligence (.ANALYSIS-Persistent-Local-Server-Intelligence.md)

The Local Intelligence analysis examines what happens when execution moves to the developer's workstation. `.github-mimic`'s high-latency issue-driven model is the complement: it optimizes for the *opposite* scenario, where execution is remote, latency is high, and each invocation must justify its cost.

A mature system would support both modes:

- **Local**: The agent runs `.github-mimic`-compiled commands on the developer's workstation with local LLM inference, zero latency, and real-time feedback.
- **Cloud**: The agent runs the same commands through GitHub Actions with full governance, auditability, and CI validation.

The command specs are portable between execution substrates. What changes is the latency, cost, and governance model—not the operational intent.

## 11. Limitations and Tensions

### 11.1 The Inference Quality Problem

`.github-mimic`'s value depends entirely on the quality of its repository inference. If the Observation Layer misidentifies the build system, misreads the test conventions, or fails to detect critical invariants, the compiled command surface will be wrong—and wrong commands are worse than no commands, because they carry an aura of authority.

The mitigation is the human-editable model: every inferred artifact is committed, reviewable, and correctable. But this places a burden on the human to validate the LLM's inference, which partially defeats the purpose of automating the understanding.

The practical response is iterative profiling: the system should generate its initial inference, present it for review, and improve through feedback cycles. Early invocations are "profiling runs" that calibrate the system's understanding. The concept document acknowledges this requirement.

### 11.2 The Maintenance Burden

A compiled command surface must be maintained. As the repository evolves—new build tools, changed test patterns, additional CI workflows—the command specs must evolve too. If re-compilation is automatic, the risk is drift: the system silently changes its own behavior in response to repository changes, potentially weakening safety gates or expanding scope without human awareness.

If re-compilation is manual, the risk is staleness: the command surface falls out of sync with the repository's actual state, producing commands that reference defunct scripts or outdated conventions.

The balance point is human-reviewed re-compilation: the system proposes updates to command specs as PRs, just as it proposes code changes. The human reviews the behavioral delta—not just the code diff, but the operational implication of the spec change.

### 11.3 The High-Latency Constraint

The concept document is honest about this limitation: "high-latency interaction makes it unsuitable for rapid back-and-forth debugging loops." This is a fundamental constraint of the GitHub Actions execution model. Each invocation takes 30 seconds to several minutes. Multi-step tasks that require iterative refinement are painful at this latency.

The Local Intelligence analysis provides the counterpoint: on a developer's workstation, the same command specs can be executed with local inference at conversation speed. The high-latency constraint is a property of the *execution substrate*, not the *command surface*. The command surface is portable; the latency depends on where it runs.

### 11.4 The Complexity Ceiling

Six lenses, twenty commands, three risk tiers, multiplexed decision trees, two-phase commits, evidence bundles—the system is not simple. The concept document describes a rich governance model that, fully implemented, would require significant engineering and significant ongoing maintenance.

The risk is that the governance overhead exceeds the value of the automation. A system that requires more effort to configure, validate, and maintain than the manual processes it replaces is a net loss.

The Fabric's response to this has always been the minimum viable principle: start with the smallest useful configuration and grow incrementally. The `.github-mimic` installation should begin with one lens, three commands, and a simple risk model. Complexity should accrete only as the system demonstrates value at each level.

---

## 12. Conclusion

`.github-mimic` proposes that repositories can compile their own operational intelligence into versioned, reviewable command surfaces. The LLM is not an assistant that answers questions—it is a compiler that observes repository reality and produces durable artifacts: command specifications, lens policies, and evidence standards that make the repository's operational knowledge explicit.

For the Fabric, this fills a specific gap. The Fabric's three planes—Source, Transformation, Execution—describe how to process code. `.github-mimic`'s Observation Layer describes how to *understand* code. Together, they form a pipeline from raw repository to governed, self-aware operational surface: observe → model → compile → execute, with governance at every boundary.

The process portability claim—that orchestration frameworks can be reduced to text and compiled into commands—is the most ambitious element, and the one most likely to encounter messy reality. But the mechanism is sound: a versioned process description, compiled into a versioned command spec, with versioned evidence at every execution. The provenance chain is complete. The behavior is auditable. The process is not hidden in a library—it is a first-class artifact in the commit graph.

The deeper implication is architectural. If `.github-mimic` succeeds, the Fabric gains the ability to ingest not just code but *operational behavior*. Upstream projects bring their mechanics with them: build systems, test patterns, release conventions, governance norms. The Fabric's Observation Layer reads them. The Compilation Layer encodes them. The Execution Plane runs them. The repository becomes not just a store of code but a self-describing operational system that knows what it does, how it does it, and what "safe change" means—expressed in versioned artifacts that humans can read, review, and correct.

The loom does not just weave threads into fabric. It learns the pattern of each thread and compiles a manual for the weaver.
