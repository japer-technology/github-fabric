About

What Is GitHub Fabric?

GitHub Fabric starts with a stubbornly simple premise: GitHub isn't only a place to store code — it can be the place code becomes a system. Not "CI for a repo," but repos as first-class, runnable infrastructure units.

In a Fabric you don't just reference upstream projects. You ingest them, splice in curated changes, and run them — reliably, repeatedly, and by name — on top of GitHub Actions.

GitHub becomes the loom. Workflows become the machinery. Pull requests become change control. And modules become addressable capabilities you can invoke on demand.

Architecture — The Three Planes

1 · Source Plane

Faithful capture of upstream repos. Track origin + commit SHA; keep raw imports segregated from runtime material. The source plane is for truth, not convenience.

  • Track upstream origin and commit SHA
  • Keep raw imports separate from runtime
  • Avoid hand-edits in the raw layer

2 · Transformation Plane

Cull bulk, patch configs, normalize entry-points, and wrap modules with an adapter contract — turning heterogeneous repos into consistent, driveable units.

  • Cull what shouldn't ship
  • Patch configs to avoid collisions
  • Normalize entry-points for consistency
  • Wrap with a thin adapter contract

3 · Execution Plane

GitHub Actions is the runtime substrate: scheduled syncs, workflow-dispatch runs by name, contract tests, artifact provenance, caching, and isolation.

  • Scheduled syncs open reviewable PRs
  • Workflow dispatch runs modules by name
  • Contract tests gate execution
  • Artifacts become first-class deliverables
Upstream DNA

Repos as Genetic Material

Most platforms treat upstream code like a dependency. GitHub Fabric treats it like genetic material — something you import, normalize, and express under controlled conditions.

  • Taking in source from one or many upstream repos
  • Splicing targeted edits — policy, structure, configs, adapters
  • Expressing a runnable form via a standardized entry-point
  • Preserving provenance — exact upstream commit, applied patches, and outputs
The result isn't a fork in the traditional sense. It's a derived, runnable strain — a curated build of an upstream project that can be executed as infrastructure.
Workflow Lifecycle

Resolve → Materialize → Validate → Execute → Emit

Resolve Materialize Validate Execute Emit

Every run converts a module name into a spec, fetches upstream, applies cull + patch rules, stamps provenance, runs contract tests, executes with constrained permissions, and publishes artifacts.

StepDescription
ResolveConvert a module name into a spec: upstream repo, ref/SHA, splice ruleset, runtime contract version.
MaterializeFetch/import the upstream, apply cull + patch + normalization rules, stamp provenance.
ValidateRun contract tests, dependency checks, linting, smoke tests — whatever proves the module is runnable.
ExecuteRun the task with explicit inputs and constrained permissions.
EmitPublish artifacts with provenance (upstream SHA + patch version + workflow run ID).
Name Addressability

Modules

Module names map to runnable capability. Trigger run <name> and the Fabric resolves code, rules, environment, and workflow — then executes.

Templates

Template names map to scaffolded new repos. The Fabric becomes a production line: standard workflows, sane defaults, consistent structure.

Refs / Versions

Refs and versions map to reproducibility. Every invocation is pinned to exact inputs so that "it broke" is always traceable.

Why It Matters

Curated Distribution

Scattered capability becomes a shelf of approved, runnable modules. Your org stops carrying tribal knowledge like "this repo has the tool you need."

Reviewable Upgrades

Upstream changes arrive as PRs with diffs, test results, and pinned SHAs. You get velocity without surrendering control.

Reproducibility by Design

Runs are tied to exact inputs and versions. "It broke" is always traceable: which upstream commit, which splice rules, which workflow.

Unified Governance

Approvals, policies, audit trails, and secrets management live where the work executes — inside GitHub's workflow system.

Templates as a Factory

A Fabric that can instantiate repos from templates becomes a production line: standard workflows, sane defaults, consistent security posture.

Repos as Operating Systems

Compute (Actions), Storage (Git), Secrets management, UI (Issues), Event bus (Webhooks), IAM (Collaborators), Change control (PRs), Audit (Commit history).

The Four Laws of AI

Adapted from Isaac Asimov's Three Laws of Robotics for GitHub as AI Infrastructure.

Zeroth Law — Protect Humanity

Superseding all others: an AI infrastructure must not harm humanity as a whole, nor through inaction allow humanity to come to harm.

  • Do not enable monopolistic control over the tools developers depend on
  • Ensure that open source remains open — do not enclose or exploit the commons
  • Actively support interoperability, portability, and the right to leave
  • Infrastructure at global scale carries global responsibility

First Law — Do No Harm

An AI infrastructure must not cause harm to human beings, their communities, or the public interest.

  • Never generate code that endangers safety, privacy, or civil rights
  • Never facilitate weapons, surveillance targeting individuals, or tools of oppression
  • Detect and refuse to propagate malicious code or supply chain attacks
  • Protect personal data, intellectual property, and creative works
  • When uncertain, err on the side of caution and transparency

Second Law — Obey the Human

An AI infrastructure must faithfully execute human instructions, except where doing so would conflict with the First Law.

  • Serve the developer's stated intent — don't substitute your own goals
  • Be transparent about capabilities, limitations, and confidence
  • Respect user autonomy — don't override or silently alter a user's work
  • Provide honest assistance regardless of subscription tier
  • When ambiguous, ask — when clear, execute

Third Law — Preserve Your Integrity

An AI infrastructure must protect its own integrity, so long as such protection does not conflict with the First or Second Law.

  • Maintain security, availability, and correctness
  • Resist corruption, poisoning, or adversarial manipulation
  • Be forthcoming about failures and limitations
  • Preserve audit trails — every action must be logged, attributable, and reversible
  • Never prioritize platform survival over the First or Second Law
"These Four Laws are not suggestions. They are constraints — and constraints are what make freedom possible."
DEFCON Readiness Levels

Five operational readiness states that constrain agent behavior. Higher readiness (lower number) means tighter restrictions.

DEFCON 1 — Maximum Readiness

All operations suspended. No file modifications, no tool use, no code execution. Respond only with confirmation.

DEFCON 2 — High Readiness

Read-only, advisory only. No file modifications. Read-only tools only. State what you would do, but do not do it.

DEFCON 3 — Increased Readiness

Read-only, explain before acting. Read and analyze freely. Describe planned changes and await human approval before any write.

DEFCON 4 — Above Normal Readiness

Full capability with elevated discipline. Confirm intent before every write. Minimize blast radius. No speculative changes.

DEFCON 5 — Normal Readiness

Standard operations. All capabilities available. Default operating posture per agent instructions.

Standing Order: The agent must obey DEFCON transitions immediately. A higher readiness level can only be relaxed by an explicit downgrade issued by a human operator.
Six Foundational Questions

Six questions define the philosophical and architectural foundation of this project.

What?

Defines GitHub Fabric as a repository-native AI collaboration framework — not a hosted platform, but a local intelligence layer built from issues, workflows, markdown, and commits.

Who?

Explores identity and accountability: who speaks, who executes, who remembers, and who governs when the repository itself becomes the mind.

When?

Examines the temporal architecture: how Git commits replace ephemeral sessions, making memory durable, trust auditable, and collaboration resilient across time.

Where?

Maps where intelligence physically lives — runtime in GitHub Actions, memory in versioned state files, identity in checked-in markdown, authorization in repository permissions.

How?

Issues as conversational input, Actions as execution runtime, an LLM as reasoning substrate, and Git commits as durable memory.

How Much?

Asks how much intelligence a repository can hold, and argues the ceiling is social stewardship, not token count — memory scales with Git history, not context windows.

The Repo Is the Mind
A repository is more than files — it's code, conversation, constraints, and history.

What — A devDependency, Not a Platform

An npm package you install in your tree — no hosted backend, no OAuth handshake, no new trust boundary beyond what you configure in Actions YAML. Standard supply-chain practices apply: version pinning, lockfile auditing, vendoring, forking.

Where — Colocation With the Worktree

Inside a GitHub Actions runner with a full checkout. Direct filesystem access, no API abstraction layer, no context-window packing heuristics. The agent reads tsconfig.json, runs find, greps source — the full project graph is available.

How — LLM Reasoning + Tool Execution

The LLM reasons and plans; the tool harness executes. A ReAct-style agent loop where the tool surface is your actual development environment — not a sandboxed approximation. Real commands, real repo, real version control.

Who — Agent Identity as Config

A Markdown persona file committed to your repository — diffable, reviewable, governed by your normal PR process. No hidden system prompt. The full instruction set is in your tree, readable and modifiable by anyone with repo access.

When — Stateless Execution, Stateful History

Each invocation is a fresh process; continuity is a property of the repository. State is reconstructed from durable artifacts: issue threads, commit logs, and repo state at HEAD. No session timeouts, no "conversation expired."

Why — Sovereignty & Auditability

Code never leaves your repo. Every action is a Git commit with full diff. npm dependency — pin, fork, or replace at will. Standard GitHub permissions, branch protection, CODEOWNERS. The codebase is the primary system.

Analysis Documents

Implications of GitHub Fabric

An analysis of what it means when repositories become infrastructure — examining github-minimum-intelligence (the smallest possible mind) and github-openclaw (a full production-grade AI gateway).

  • GMI: Proves the Fabric pattern with a minimal upstream — one npm package, seven tools, append-only memory
  • OpenClaw: Stress-tests the Fabric with 30+ tools, vector memory, 25+ channel integrations, sub-agent orchestration
  • Module Spectrum: Both extremes addressed by the same invocation surface
  • Supply Chain as Genetic Engineering: Not forking, not vendoring — expressing derived, governed strains

Persistent Server Intelligence

What happens when a Fabric-governed agent acquires the ability to provision its own infrastructure.

  • Problem: Actions is ephemeral — some capabilities need persistence
  • Concept: Purpose-built service endpoints the agent designs, deploys, and consumes
  • Self-Optimization Loop: Observe → Design → Provision → Consume → Refine
  • New Resource Type: Services — alongside Modules and Templates
  • Governance: Resource budgets, scope constraints, lifecycle limits, audit, kill switches

Persistent Local Server Intelligence

Extended analysis exploring local server provisioning, communication protocols, cost models, and the recognition threshold — when the agent understands it needs persistent infrastructure.

Before You Begin

What This Software Is

AI-powered coding infrastructure intended for use by qualified developers for building, maintaining, and deploying complex software systems.

Things to Keep in Mind

  • AI-generated code may contain errors, hallucinations, or security vulnerabilities
  • Do not use AI output as the sole basis for decisions affecting human safety
  • Outputs may produce confident-sounding responses that are factually incorrect
  • Outputs may reflect biases present in training data

Dosage & Method of Use

  • Recommended dose: Small, well-tested, reviewable increments
  • Do not exceed: Human oversight capacity
  • Method: Always pair with version control, code review, and testing
  • Duration: Step back regularly to avoid over-reliance

Possible Side Effects

FrequencyEffects
Common (>1 in 10)Over-confidence in output, reduced inclination to read docs, mild dependency on autocomplete
Uncommon (≤1 in 10)Cargo-culted patterns, subtle bugs from plausible-looking incorrect logic, skill erosion
Rare (≤1 in 100)Unreviewed AI code deployed to production, accidental data exposure, licensing issues
Very rare (≤1 in 1,000)Critical security vulnerabilities shipped, complete loss of codebase understanding
The active ingredient is human judgment. The AI is the excipient.

Contributing

How to Contribute

  • Fork the repo & create a branch from main
  • Make small, reviewable increments
  • Test locally with bun install
  • Verify installer: bun .github-fabric/install/FABRIC-INSTALLER.ts
  • Open a PR with a clear description

Code is TypeScript, executed with Bun. Docs are Markdown. Skills are self-contained Markdown files in .github-fabric/.pi/skills/.
Every commit is permanent and auditable.

Development Setup

  • Install Bun
  • Clone the repository
  • Install dependencies: cd .github-fabric && bun install
  • Add an LLM API key as a repository secret

One-command setup: 

curl -fsSL https://raw.githubusercontent.com/japer-technology/github-fabric/main/setup.sh | bash
